## Tool Inventory

**Tool 1: search_listings**

**Purpose:** Searches the thrift listings dataset and returns clothing items that best match the user's request. 

**Inputs:**
- description (str): Natural language description of the desired item
- size (str | None): Optional size filter; if None, size is ignored
- max_price (float | None): Optional maximum price; if None, price is ignored

**Output:**
- list[dict]: A list of matching listing dictionaries sorted by relevance; Each dictionary contains fields such as id, title, description, category, style_tags, size, condition, price, colors, brand, and platform

**Tool 2: suggest_outfit**

**Purpose:** Uses the selected thrift item together with the user's wardrobe to generate one or more outfit suggestions 

**Inputs:**
- new_item (dict): The selected listing returned from search_listings
- wardrobe (dict): The user's wardrobe dictionary containing an items list

**Output:**
- str: A natural-language outfit recommendation; If the wardrobe is empty, the tool returns general styling advice for the selected item

**Tool 3: create_fit_card**

**Purpose:** Generates a short, social-media-style caption describing the thrifted item and suggested outfit 

**Inputs:**
- outfit (str): Outfit suggestion generated by suggest_outfit
- new_item (dict): The selected listing dictionary

**Output:**
- str: A casual caption suitable for sharing; If the outfit string is empty or invalid, the tool returns an informative error message instead of failing

## Planning Loop

FitFindr follows a sequential planning loop with conditional branching based on tool outputs. 

1. The agent receives the user's natural-language query.
2. The query is parsed to identify a description, optional size filter, and optional maximum price. 
3. The agent calls search_listings using the parsed values.
4. If no listings are returned, the agent immediately stops execution, stores an error message in the session state, and returns a helpful response to the user.
5. If results are found, the highest-ranked listing is selected and stored as the active item. 
6. The selected item and the user's wardrobe are passed to suggest_outfit to generate styling recommendations.
7. The resulting outfit suggestion is passed to create_fit_card, which produces a shareable caption.
8. The completed session state is returned and displayed in the interface.

The agent does not call every tool unconditionally. Instead, it makes decisions based on intermediate results and exits early when appropriate.

## State Management

FitFindr maintains a session dictionary throughout each interaction to share information between tools. 

The session stores:
- query: the original user request
- parsed: extracted search parameters
- search_results: all matching listings returned by search_listings
- selected_item: the top-ranked listing chosen for styling
- wardrobe: the wardrobe provided for the interaction
- outfit_suggestion: the output of suggest_outfit
- fit_card: the caption generated by create_fit_card
- error: an error message if execution terminates early

Using this shared session object ensures that downstream tools receive the exact outputs produced by previous steps without recomputing information. 

## Error Handling

FitFindr includes explicit handling for the primary failure mode of each tool. 

- search_listings: If no matching items are found, it returns an empty list rather than raising an exception. The planning loop detects this condition, sets an informative error message, and stops before calling additional tools. 
   - Example Tested: Searching for "designer ballgown" with size "XXS" and a maximum price of $5 returned [], after which the agent displayed an error message instructing the user to broaden their search. 
- suggest_outfit: If the wardrobe contains no items, it generates general styling advice instead of attempting to reference nonexistent clothing pieces. 
- create_fit_card: If the outfit description is empty or missing, it returns the message "Unable to generate fit card because outfit information is missing. Try generating an outfit first." rather than raising an exception. 

These safeguards allow the overall agent to fail gracefully and continue providing meaningful feedback to the user. 

## Spec Reflection

The planning specification was valuable because it defined the interfaces and responsibilities of each tool before implementation, making it straightforward to develop and test each component independently and then connect them through the planning loop. 

One divergence from the original plan is that search_listings ranks results primarily using simple keyword overlap after applying size and price filters, rather than using a more sophisticated scoring method that also considers factors such as price closeness or weighted style-tag matching. This simplified implementation was sufficient for the mock dataset while keeping the code easier to understand and debug.
